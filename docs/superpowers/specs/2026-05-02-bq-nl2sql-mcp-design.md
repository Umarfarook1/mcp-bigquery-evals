# mcp-bigquery-evals - Design Spec

**Status:** Approved 2026-05-02
**Owner:** Umarfarook Gurramkonda
**Target ship:** v1.0 in 7–10 working days
**Project context:** Project 1 of 5 in the credibility portfolio (`memory/project_portfolio_plan_2026.md`)

---

## 1. Overview

An MCP (Model Context Protocol) server that lets AI coding agents - primarily Claude Code, Cursor, and Claude Desktop used as a developer tool - explore and query a BigQuery warehouse safely. The server exposes seven read-only tools, enforces per-query cost caps via dry-run estimation, and ships with a built-in NL→SQL evaluation harness using result-set equivalence methodology.

**One-line pitch:** *An MCP server for BigQuery exploration with cost guardrails and a built-in NL→SQL eval harness.*

## 2. Goals

1. Make it possible for an AI coding agent to answer warehouse questions ("how many users signed up last week?") inside the dev workflow, without melting the BQ bill.
2. Ship measurable quality - every release has an `accuracy` number on the README, generated by an eval harness anyone can re-run.
3. Be installable in under 5 minutes by anyone with a GCP project and `uvx`.
4. Establish a credibility artifact (deployed PyPI package + README badge + `awesome-mcp-servers` listing + 2 blog posts).
5. Seed the eval harness module that becomes Project 2 (standalone NL2SQL leaderboard).

## 3. Non-goals (v1)

- ❌ Write operations (INSERT/UPDATE/DELETE/CREATE)
- ❌ An `explain_query` tool - circular when the consumer is itself an LLM
- ❌ A semantic layer or metric definitions (Project 5 territory)
- ❌ A web UI
- ❌ Multi-warehouse abstraction (Snowflake/Postgres/etc.)
- ❌ LLM-as-judge methodology in evals
- ❌ Internal LLM calls anywhere in the server runtime

## 4. Target users

Primary: AI coding agents (Claude Code, Cursor, Claude Desktop *as developer tool*). The agent chains queries, tolerates structured JSON errors, and composes the seven low-level tools to answer warehouse questions.

Out of scope for v1 (deferred to potential v2): non-technical operators using Claude Desktop as a BI front-end. That persona requires plain-English answer composition, chart-ready outputs, and stricter refusal behavior - a different product surface.

## 5. Architecture

### 5.1 BigQuery client abstraction

The single most important architectural decision. `BigQueryClient` is a `Protocol` with two implementations:

```python
class BigQueryClient(Protocol):
    def list_datasets(self) -> list[Dataset]: ...
    def list_tables(self, dataset_id: str) -> list[Table]: ...
    def get_table(self, table_id: str) -> Table: ...
    def sample_rows(self, table_id: str, n: int) -> list[dict]: ...
    def dry_run(self, sql: str) -> DryRunResult: ...
    def execute(self, sql: str) -> QueryResult: ...
```

- `RealBigQueryClient` - wraps `google.cloud.bigquery.Client`; used in production and integration tests.
- `FakeBigQueryClient` - in-memory; loaded from `tests/fixtures/fake_warehouse.yaml`; used in unit tests and during development before GCP credentials are wired.

All tool handlers depend on the `Protocol`, never the concrete class. This decouples server development from GCP setup and keeps the unit-test loop fast (no network, no credentials, no quota burn).

### 5.2 Tool layer

Each MCP tool lives in its own file under `src/mcp_bigquery_evals/tools/`. A tool is a thin handler that:
1. Validates input
2. Calls one or more `BigQueryClient` methods
3. Shapes the result as MCP-tool-response JSON
4. Returns

No state. No side effects beyond the BQ call itself. No internal LLM calls.

### 5.3 Server entrypoint and CLI surface

A single binary (`mcp-bigquery-evals`) with two subcommands:

- `mcp-bigquery-evals serve` (default if no subcommand given) - registers the seven tools with the MCP Python SDK and runs a stdio MCP server. This is what `claude_desktop_config.json` invokes.
- `mcp-bigquery-evals evals run --model <model> [--limit N]` - runs the eval harness; not used by the MCP runtime; intended for CI and local benchmarking.

Both subcommands wire up `BigQueryClient` the same way: if `MCP_BIGQUERY_FAKE_FIXTURE` is set → use `FakeBigQueryClient` loaded from that yaml path; otherwise → use `RealBigQueryClient` with ADC + `BIGQUERY_PROJECT`.

## 6. MCP tool surface (v1)

All seven tools are read-only.

| Tool | Input | Output | Purpose |
|---|---|---|---|
| `list_datasets` | - | `[{id, location, description}]` | Discovery of datasets in the configured project |
| `list_tables` | `dataset_id: str` | `[{id, row_count, size_bytes}]` | Discovery of tables within a dataset |
| `describe_table` | `table_id: str` | `{columns: [{name, type, description}], row_count, size_bytes}` | Schema + metadata for grounding |
| `sample_table` | `table_id: str, n: int = 5` | `[rows]` | Returns sample rows so the agent can see real values before writing SQL |
| `search_schema` | `term: str` | `[{table, column, similarity}]` | Fuzzy-match a natural-language term to actual schema identifiers (the differentiator) |
| `estimate_cost` | `sql: str` | `{bytes_scanned, estimated_usd}` | Free dry-run; gates `run_query` |
| `run_query` | `sql: str, max_bytes_scanned: int = 100_000_000` | `{rows, bytes_scanned, cost_usd, ms}` | Executes after dry-run check; auto-rejects if over cap |

## 7. Cost guardrail mechanism

- Every `run_query` call dry-runs first (free; BQ returns bytes-to-be-scanned in milliseconds).
- If `bytes_scanned > max_bytes_scanned`, `run_query` returns a structured error:
  ```json
  {
    "error": "cost_cap_exceeded",
    "would_scan": "1.2 GB",
    "cap": "100 MB",
    "estimated_usd": 0.006,
    "hint": "narrow your WHERE clause or pass max_bytes_scanned=2_000_000_000"
  }
  ```
- Default cap: **100 MB scanned** (≈ $0.0005 at on-demand pricing).
- Cap is per-call; the agent may raise it explicitly per query. No hidden global cap in v1.
- `MCP_BIGQUERY_GLOBAL_DAILY_LIMIT_USD` env var is reserved for v1.1; not implemented in v1.

## 8. Eval harness

| Aspect | Decision |
|---|---|
| **Format** | `evals/golden.yaml` - list of `{id, dataset, nl, gold_sql, tags}` |
| **Methodology** | Result-set equivalence (Spider/BIRD-style): execute both gold SQL and model SQL, compare result sets as multisets of rows (order-independent), pass = identical results |
| **Initial size** | 30–50 pairs across 3–5 `bigquery-public-data` datasets (target: `stackoverflow`, `new_york_taxi_trips`, `github_repos`) |
| **CLI** | `mcp-bigquery-evals evals run --model claude-haiku-4-5 [--limit N]` (subcommand on the same entrypoint binary as the server) |
| **Output** | JSON report: `{accuracy, per_pair_results, avg_bytes_scanned, avg_latency_ms, total_cost_usd}` |
| **CI integration** | Eval suite runs against fixtures on every PR; runs against real BQ on `main` merges (once GCP credentials are wired in repo secrets) |
| **README badge** | Live `accuracy` percentage per model, auto-updated by CI |
| **Models initially benchmarked** | `claude-haiku-4-5`, `claude-sonnet-4-6` |
| **How models are called** | Eval runner uses the Anthropic Python SDK directly (`anthropic.Anthropic().messages.create(...)`). The runner is independent of the MCP server runtime - it loads `BigQueryClient` directly to execute gold and predicted SQL for result comparison. The MCP server itself never calls any LLM. |

The harness lives at `src/mcp_bigquery_evals/evals/` for v1. When Project 2 begins, this module is extracted into a standalone PyPI package (`nl2sql-evals`); `mcp-bigquery-evals` then imports it. This preserves the spinoff narrative without double-building.

## 9. Distribution & authentication

- **Package:** `mcp-bigquery-evals` on PyPI
- **Install/run:** `uvx mcp-bigquery-evals`
- **End-user wiring** in `claude_desktop_config.json`:
  ```json
  {
    "mcpServers": {
      "bigquery-evals": {
        "command": "uvx",
        "args": ["mcp-bigquery-evals"],
        "env": { "GOOGLE_APPLICATION_CREDENTIALS": "/path/to/sa.json" }
      }
    }
  }
  ```
- **Auth detection:** if `GOOGLE_APPLICATION_CREDENTIALS` is set → use service account JSON; otherwise → fall back to ADC (Application Default Credentials, populated by `gcloud auth application-default login`).
- **Project selection:** `BIGQUERY_PROJECT` env var; required.

## 14. Decisions and rationale (interview-defensible)

| Decision | Rationale |
|---|---|
| Python over TypeScript | `google-cloud-bigquery` is the canonical client; mature pytest tooling for the eval harness; aligns with Project 5 stack. |
| `uvx` over `pip install` over Docker | Adoption friction is the #1 killer of OSS projects; `uvx` is zero-install; Docker is too much tax for casual try-it-out. |
| Read-only v1 (no INSERT/UPDATE/DELETE) | Eliminates an entire class of guardrail bugs; makes v1 a 7–10 day ship instead of 3 weeks; easy to extend in v1.x. |
| `BigQueryClient` as Protocol | Decouples server development from GCP setup; lets the in-memory fake unblock day-1 dev; standard Python pattern (PEP 544). |
| Result-set equivalence over LLM-as-judge | Matches Spider/BIRD academic standard; deterministic; respected by senior-engineer audience (Hamel Husain has written explicitly against LLM-as-judge for SQL). |
| Cost cap via mandatory dry-run | BQ dry-run is free and returns bytes-scanned in ms; eliminates the worst failure mode (agent runs a 1 TB scan); structured error makes the agent self-correct. |
| Dropped `explain_query` tool | The MCP consumer is Claude - it can summarize its own SQL. Putting an LLM inside an MCP server whose only consumer is an LLM is a circular design smell. |
| Default cap 100 MB | ≈ $0.0005 per query; covers ~95% of legitimate exploration queries on `bigquery-public-data`; agent can raise per-call. |
| MIT license | Maximum adoption surface; no patentable inventions in v1; compatible with all downstream uses. |
| Single repo for MCP + evals (v1) | YAGNI - extract to two packages when Project 2 begins, not before; spinoff narrative works either way. |
